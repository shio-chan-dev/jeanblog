# Claim Boundary Checklist

Run this pass before finalizing the article.

## Verified Facts

These should be phrased as facts only when the author actually verified them:

- observed product behavior
- commands that were run
- repository structure or code paths
- documented product capabilities from official docs

## Personal Inference

These are valid, but should sound like judgment rather than fact:

- "I think..."
- "In my workflow..."
- "For teams that already have..."
- "My conclusion after trying it is..."

## Open Questions

If the author did not verify something, keep it explicit:

- "I did not test..."
- "I have not verified whether..."
- "This may change if..."

## Red Flags

Rewrite or split the claim if the draft does any of these:

- turns one local trial into a universal recommendation
- treats product marketing language as established fact
- hides a weak evidence base behind a strong conclusion
- confuses "useful for memory retrieval" with "safe as a source of truth"
