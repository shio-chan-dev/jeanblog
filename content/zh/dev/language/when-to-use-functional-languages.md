## 副标题 / 摘要

函数式语言擅长处理并发与复杂逻辑，但并非所有场景都适用。本文给出清晰的适用边界。

## 目标读者

- 进行语言选型的团队
- 想理解函数式价值的工程师
- 关注可维护性的技术负责人

## 背景 / 动机

函数式编程提高可推理性，但学习成本与性能特性也带来代价。  
明确适用场景有助于合理选型。

## 核心概念

- **纯函数**：无副作用，便于测试
- **不可变性**：并发友好
- **表达力**：复杂逻辑更清晰

## 实践指南 / 步骤

1. **并发场景优先考虑函数式**
2. **业务规则复杂时优先使用纯函数**
3. **I/O 密集系统需评估生态与性能**
4. **团队学习成本纳入评估**

## 可运行示例

```python
# 函数式风格更适合复杂规则组合

def apply_rules(x, rules):
    for r in rules:
        x = r(x)
    return x


if __name__ == "__main__":
    print(apply_rules(10, [lambda x: x + 1, lambda x: x * 2]))
```

## 解释与原理

函数式适合“规则多、并发高、可推理性强”的场景。  
但在高性能或生态依赖强的场景要谨慎评估。

## 常见问题与注意事项

1. **函数式一定更安全？**  
   更易推理，但仍需正确设计。

2. **是否适合所有团队？**  
   不一定，学习成本较高。

3. **性能会不会更差？**  
   取决于实现与数据结构。

## 最佳实践与建议

- 先在模块中试点
- 评估生态与性能指标
- 在团队内建立函数式规范

## 小结 / 结论

函数式语言适合复杂逻辑与并发场景，但不适合所有系统。  
正确的选型需要结合业务与团队能力。
