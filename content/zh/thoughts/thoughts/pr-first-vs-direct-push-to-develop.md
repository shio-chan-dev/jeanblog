> 核心结论：如果一个改动会进入协作、验收、发布说明或后续追踪，我现在更倾向于让它先经过 `feature branch -> PR -> develop`。直接往 `develop` 推并不是永远错误，但它更适合低风险、个人化、短生命周期的小改动，不适合作为默认团队工作流。

## 目标读者

- 已经在用 `develop` 分支，但还在犹豫要不要把 PR 变成默认入口的开发者
- 会写 release notes、changelog 或交付说明的小团队负责人
- 想把“提交代码”和“形成可追踪交付记录”分开的工程师

## 背景与动机

我这次并不是先从 Git 分支模型开始思考，而是从一个更具体的问题倒推：

**为什么很多发布说明更自然地引用 PR，而不是逐条引用 commit hash？**

当我顺着这个问题往下想时，工作流边界就慢慢清楚了。

如果一条发布说明通常对应的是：

- 一个功能切片
- 一次可评审的改动
- 一组有上下文的讨论
- 一次带验收结果的合并

那它天然更接近 PR，而不是单个 commit。

这也逼着我重新判断一个看起来很日常的问题：

**每加一个功能，到底应该直接推到 `develop`，还是应该先走 PR？**

## 核心概念

### 1. Commit

commit 适合描述一次代码变化。它回答的是：

- 这一小步代码改了什么？
- 为什么改？

但它未必天然适合作为对外协作单位。因为一个功能通常不只是一条 commit。

### 2. PR

PR 更像一个“可审查、可讨论、可验收的交付切片”。它回答的是：

- 这个功能范围是什么？
- 改动是不是该合并？
- review 过程中有哪些补充判断？

它不只是代码差异，也是一次协作记录。

### 3. develop 分支

`develop` 更适合作为“准备集成”的主线，而不是每个原子动作的直接落点。

一旦大家都直接往 `develop` 推，它当然会变快，但也会更难区分：

- 哪个功能还在半成品状态
- 哪个改动已经被 review 过
- 哪个版本说明应该引用哪组变更

如果你的仓库里还有 `master` 或 `main` 作为正式发布线，那边界会更清楚：

- `feature branch` 承载具体开发
- `develop` 承载日常集成
- `master` 或 `main` 承载准备打 tag 的发布结果

这时候 PR 也不应该都直接打到 `master`。大多数功能更自然的路径仍然是：

`feature -> PR -> develop`

只有当一批改动已经形成一个准备发布的版本切片时，才再由 `develop` 进入 `release` 或 `master`。

### 4. Release Notes

发布说明的工作，不是罗列仓库里发生过什么，而是整理出：

- 哪些变化值得对外说明
- 这些变化该按什么单位分组
- 后续读者应该从哪里追溯上下文

从这个角度看，PR 往往比 commit 更像 release notes 的自然来源。

## 我原本以为直接推 develop 也没什么问题

在很多小项目里，直接推 `develop` 看起来确实没什么不行：

- 少一个步骤
- 不用开分支
- 不用写 PR 描述
- 自己一个人时也不一定需要 review

如果只看“今天能不能把代码传上去”，它甚至是最省事的。

所以我一开始真正犹豫的点不是“PR 对不对”，而是：

**PR 到底是不是值得成为默认动作，而不是一种团队变大之后才需要的额外仪式？**

## 我后来真正看清的是：发布说明反而暴露了工作流问题

这次让我改变判断的，不是某个 Git 理论，而是 release notes 的组织方式。

当你准备写一版正常的发布说明时，你通常不会想写成这样：

- commit `a1b2c3`
- commit `d4e5f6`
- commit `g7h8i9`

你更可能想写成：

- 新增了什么功能
- 修了什么问题
- 哪个改动对应哪个 PR
- 完整变更范围的 compare link 在哪里

也就是说，发布说明天然偏好“有边界的改动单元”，而不是“仓库历史里所有原子提交”。

这时我就意识到：

如果我的日常开发默认就是“功能分支 -> PR -> 合并”，那发布说明、变更回溯、协作讨论，几乎都会顺着同一条追踪链往下走。

但如果默认是“想到哪改到哪，然后直接推 develop”，那后面很多环节都会开始补救：

- 得自己手动还原一个功能边界
- 得从多条 commit 里拼出一条发布说明
- 得回翻聊天记录找当时为什么这么做

## 它适合做什么

这里我按几个真正影响判断的维度来比较。

### 1. 追踪性：PR 比直接推 develop 更适合作为变更单位

我的旧直觉是：commit 已经可以追踪了，为什么还要多一层 PR？

后来我发现，两者追踪的不是同一个层级。

- commit 追踪的是“代码怎么一步步变成现在这样”
- PR 追踪的是“这组改动为什么被当成一个功能或修复被合进去”

如果你后面要写 release notes、做回顾、看验收记录，PR 的粒度通常更接近你真正需要的粒度。

我的判断是：**只要改动会进入对外说明或协作交付，PR 通常比单条 commit 更像自然的追踪单位。**

### 2. 验收性：PR 天然区分了生产和验收两个阶段

你前面提到一个很关键的判断法：很多事情值得拆成“生产”和“验收”两个阶段。

我现在觉得 PR workflow 本身就是这个原则在工程交付里的体现。

- 分支上的开发，是生产阶段
- PR 的 review、讨论、通过与否，是验收阶段

如果直接推 `develop`，这两个阶段会更容易混在一起。代码当然还是能交付，但质量闸门不够清楚。

我的判断是：**只要你认为“写完”和“可合并”不是一回事，PR 就不是多余动作。**

### 3. 发布说明适配度：PR 比 commit 更容易生成面向读者的说明

release notes 关心的是：

- 用户或协作方能感知到什么变化
- 这次发布里有哪些功能或修复
- 如果需要回溯，应该点哪里

大多数情况下，PR 编号、PR 链接，甚至 PR 标题，都比 commit hash 更适合直接暴露给读者。

这不代表 commit 不重要。commit 仍然是底层真实历史。

但从“给人看”的层面说，PR 更像是“已经整理过一层的变更入口”。

我的判断是：**如果你希望发布说明写得像给人看的版本记录，而不是 git log 摘抄，PR-first workflow 会更顺手。**

### 4. 团队协作适配度：PR 更容易形成默认纪律

一个人单干时，很多规范都可以靠自觉。

但只要进入多人协作，默认动作就开始决定系统质量。

PR 的好处不是“形式更正规”，而是它天然要求团队回答几个问题：

- 这次改动边界是什么？
- 谁来 review？
- 合并前有没有明显风险？
- 这条改动之后怎么进入发布说明？

这些问题如果不在 PR 阶段回答，后面通常也还得回答，只是代价更高。

### 5. 大功能拆分适配度：PR 的单位不应该等于整个 feature

这也是我后来才真正想清楚的一点：

**一个 feature 不一定只对应一个 PR。**

很多混乱其实不是因为用了 `develop`，而是因为把一个很大的 feature 全部塞进了一个分支，然后默认它应该对应一个同样巨大的 PR。

我现在更接受的判断是：

**一个 PR 的单位，应该是一个清晰、可审查、可回滚的变更切片，而不是“这个 feature 目前所有做过的东西”。**

如果一个 feature 很大，更适合的做法通常是把它拆成几个阶段性的子 PR，例如：

- 先补基础数据结构或接口边界
- 再补核心逻辑
- 再补 UI 或接入层
- 最后补测试、权限和收尾

如果能按用户可感知的能力纵向拆分，就更好。因为这样每个 PR 都更像一个小功能，而不是技术层碎片。

## 它不适合做什么

我并不想把结论写成“所有情况都必须 PR”。

### 1. 不适合把极小、纯个人、短生命周期改动也强行流程化

例如：

- 自己的临时实验
- 很小的 typo 修正
- 一次性草稿
- 根本不会进入 release notes 的本地验证

这类改动如果也强制开分支、提 PR、等完整 review，有时只是把工作流变重。

我的判断是：**PR 适合做默认主路径，但不应该变成没有例外的宗教规则。**

### 2. 不适合把 PR 当成 source of truth

这是另一个很容易走偏的地方。

PR 很适合做：

- 协作入口
- 验收记录
- 发布说明的上游输入

但它不应该替代：

- issue 对范围和目标的定义
- 正式文档对稳定结论的记录
- commit 对底层实现演化的记录

换句话说，**PR 是交付和验收层，不是全部知识都要堆进去的地方。**

### 3. 不适合用来掩盖拆分不清的问题

有些团队会以为“只要提了 PR，就自动变规范”。

其实不是。

如果一个 PR 又大又杂、功能边界混乱、描述含糊，那它只是把混乱集中到了另一个界面里。

所以真正重要的不是“有没有 PR”四个字，而是：

- PR 是否对应清晰的改动单元
- 是否真的经过验收
- 是否能被后续 release notes 正常消费

### 4. 不适合把“先开 PR”误解成“最后一定只有一个大 PR”

如果一个 feature 已经很大，而你只是把所有改动堆在一个分支里，最后再统一发一个 PR 到 `develop`，那本质问题并没有消失。

这时 PR 只是把混乱换了一个入口。

我现在更倾向于把它理解为：

- PR-first 解决的是“交付和验收的边界”
- PR 拆分解决的是“改动粒度和 review 成本”

这两个问题不能互相替代。

## 和我当前工作流的边界

我现在更接受的边界是这样的：

- `issue` 负责定义为什么做、范围是什么、验收标准是什么
- `feature branch` 负责承载生产阶段
- `PR` 负责承载验收阶段
- `develop` 负责承载已经准备好集成的结果
- `release` 或 `test` 分支负责承载一批准备展示、验证或发布的集成结果
- `master` 或 `main` 负责承载准备打 tag 的正式发布线
- `release notes` 负责把已合并结果重新整理成对外可读的说明

这套边界最打动我的地方，不是“更像大团队流程”，而是每一层的职责终于更清楚了。

直接推 `develop` 最大的问题不是一定会坏，而是它容易让这些边界全部塌在一起。

如果是单人开发，我现在觉得最容易失控的不是“没有人 review”，而是：

- 一个大 feature 在一个分支里越积越多
- `develop` 变成临时堆放所有半成品的地方
- 到要发版时才发现很难解释哪些东西该一起进 release

所以哪怕是单人项目，PR-first 也未必只是为了“别人 review 我”，它也可能是在帮未来的自己做结构化验收。

## 如果一个 feature 已经很大，应该怎么拆

我现在更认同一个很实用的原则：

**一个 feature 可以拆成多个 PR，但每个 PR 最好仍然表达一个清晰的交付切片。**

具体来说，可以优先按下面几种方式拆：

### 1. 按阶段拆

例如：

- 基础模型或接口
- 核心业务逻辑
- UI 或集成接入
- 测试与边界补充

这种拆法的好处是自然，缺点是有时会让中间 PR 过于技术化。

### 2. 按用户可感知能力拆

例如不是拆成：

- 后端接口
- 前端页面
- 测试

而是拆成：

- 支持创建草稿
- 支持编辑草稿
- 支持提交草稿
- 补权限和边界测试

这类拆法通常更利于 review，也更利于后面生成 release notes。

### 3. 按依赖关系做 stacked PR

如果几个改动前后依赖明显，那也没必要硬拆成完全独立的平行 PR。

这时更好的做法可能是：

- PR A 先引入基础能力
- PR B 基于 PR A 增加下一层能力
- PR C 再基于前两者补完整流程

这样每一层 review 的焦点会更清楚。

### 4. 中间状态不能进 develop 时，再考虑 feature 集成分支

如果一个大 feature 的中间状态根本不能合入 `develop`，那可以先用一个 feature 集成分支做过渡，例如：

- `feature/reporting-base`
- `feature/reporting-ui`
- `feature/reporting-export`

这些小 PR 先合到一个集成分支，等达到可集成状态后，再统一进入 `develop`。

但这通常是次优解。能直接把小 PR 合进 `develop` 并用 feature flag 隐藏未完成入口时，长期维护成本往往更低。

## 如果分支已经乱了，怎么补救

这也是我最近更在意的一个现实问题：很多时候混乱不是从今天开始的，而是已经在一个分支里堆了很多改动。

如果已经发生这种情况，我现在更倾向于这样处理：

### 1. 先停止继续往当前大分支里加东西

不要一边整理一边继续扩散范围。先把现有改动视为一堆“待切分素材”。

### 2. 按功能或阶段重新定义边界

先问自己：

- 哪些改动其实属于同一个能力？
- 哪些改动只是顺手带上的？
- 哪些改动值得作为一个单独 PR 存在？

### 3. 如果 commit 已经相对干净，就用 `cherry-pick` 拆出去

这是我现在觉得很实用的补救手段。

如果原分支上的改动已经能分成若干独立 commit，那就可以：

- 从 `develop` 拉出新的小分支
- 把对应 commit `cherry-pick` 过去
- 分别发 PR

这样比把一整个大分支直接提成一个 PR 更容易救回来。

### 4. 如果 commit 本身就混乱，先整理 commit，再拆分支

`cherry-pick` 很适合搬运已经相对独立的 commit，不适合直接处理一团混在一起的改动。

所以如果原始 commit 已经把多个功能揉在一起，真正的第一步往往不是发 PR，而是先把 commit 整理到可以被独立挑出来。

### 5. 把旧分支当“素材分支”，不要再把它当长期主开发线

这一步非常重要。否则你会永远在一个已经失去边界的分支上继续补丁式工作。

## alpha 版本应该怎么配合 PR

我现在也不太认同“每合完一个 feature 就机械地发一个 alpha”。

更准确的说法应该是：

**alpha 应该对应一个可展示、可反馈的阶段性集成点，而不是每一个小 PR。**

比较适合发 alpha 的情况通常是：

- 一组 PR 已经形成了一个能演示的能力
- 主流程已经跑通，哪怕还不完整
- 你需要让别人看演进、收反馈或做阶段验收

不太适合每次都发 alpha 的情况通常是：

- 只是内部重构
- 只是基础设施准备
- 一个小 PR 本身还不足以形成可展示能力

所以我现在更接受的节奏是：

- 小 PR 持续合到 `develop`
- 一批 PR 形成阶段性成果时，再发一个 `alpha`
- 稳定后再进入正式 release 流程

这样 alpha 更像“版本里程碑”，而不是“每个功能都得打一次版本号”。

## 一个给读者的最小判断法

如果你也在犹豫要不要把 PR 变成默认动作，我建议先问四个问题：

1. 这个改动会不会进入 release notes 或 changelog？
2. 这个改动是否需要别人 review、验收或继续接手？
3. 一周后回头看时，你是否需要一个比 commit 更高层的追踪单位？
4. 这个改动失败时，是否值得单独定位为一个“被合并过的功能切片”？

如果有两个以上答案是“会”，那我会建议你优先走 PR。

如果四个答案几乎都是“不会”，那直接推个人分支甚至直接提交到主开发线，往往也不是问题。

如果你已经有 `master`、`develop`、`release` 这类分层，我觉得还可以再补两个问题：

5. 这个改动是否已经到了“准备集成”，而不只是“本地完成”的程度？
6. 这个 feature 是否大到应该拆成多个 PR，而不是硬塞进一个分支？

只要第 5 个问题答案是否定的，就还不该急着进 `develop`。
只要第 6 个问题答案是肯定的，就应该优先拆 PR，而不是争论“到底 direct push 还是 PR”。

## 解释与取舍

这篇文章不是在说 direct push 完全错误。

更准确的说法是：

- direct push 追求的是操作最短路径
- PR-first 追求的是可评审、可追踪、可发布说明化

两者都能把代码放进仓库，但它们优化的目标并不一样。

在我的当前判断里，只要一个功能已经开始碰到“协作、验收、发布”这三个词，PR-first 的收益通常就会超过它的流程成本。

如果再往前多走一步，我现在会把它总结成两层判断：

- 第一层：这个改动是否值得经过独立的验收入口？
- 第二层：这个改动是否已经被拆成了合适的 PR 粒度？

很多所谓的工作流混乱，其实不是只缺第一层，而是两层都没有建立起来。

## 常见问题与注意事项

### PR 一定比 direct push 慢吗？

表面上通常是更慢一点，但很多时候它只是把本来会在后面返工的解释和补救，提前到了合并前。

### 单人项目也值得用 PR 吗？

不一定。如果你只是快速试验，未必值得。但只要你开始需要更稳定的版本边界、发布说明或未来回看记录，单人项目也可能受益。

### 如果 `master` 是发布分支，还应该每个功能都直接往 `master` 发 PR 吗？

通常不应该。

如果 `master` 进入后就意味着准备打 tag，那更自然的路径往往是：

- `feature -> develop`
- `develop -> release` 或 `test`
- `release` 或 `test` -> `master`

这样 `master` 前的 PR 更像一次版本发布，而不是一堆开发中的功能片段。

### 一个 feature 很大，是否就只能对应一个大 PR？

不是。

更好的理解是：一个 PR 对应一个清晰、可审查、可回滚的变更切片。一个大 feature 完全可以拆成多个子 PR。

### 如果我已经在一个大分支里做了很多散乱改动，还来得及救吗？

来得及。

如果 commit 还算干净，可以通过 `cherry-pick` 把不同能力挑到新分支，再分别发 PR。真正难救的不是分支多，而是 commit 自身已经失去边界。

### 每合完一个 feature 都应该发一个 alpha 吗？

不一定。

更适合发 alpha 的时机，是一组 PR 已经形成了一个可展示、可反馈的阶段性成果，而不是每个小 feature 一合并就立即升一个 alpha。

### release notes 一定要带 commit hash 吗？

不一定。大多数项目更常见的是带 PR 或 issue 引用，再加一个完整 changelog compare link。commit hash 更适合做底层追溯，不一定适合作为公开说明的主入口。

## 最佳实践与建议

- 把 PR 当成默认的功能交付入口，而不是所有改动都必须遵守的铁律
- 如果 `master` 是发布线，就让日常功能优先通过 `feature -> PR -> develop`
- 一个大 feature 不要默认只配一个大 PR，优先拆成多个可审查切片
- 如果分支已经变乱，先停手整理，再用 `cherry-pick` 把可独立的改动搬到新分支
- 把 alpha 版本当阶段性集成里程碑，而不是每个小功能的自动动作
- 让 release notes 优先引用 PR 或 issue，而不是逐条罗列 commit
- 让 `develop` 承载“准备好集成”的结果，而不是所有中间状态
- 保留例外通道，但只给极小、低风险、不会进入正式发布记录的改动

## 结论

我的最终判断是：

**如果一个功能未来要被 review、被发布、被追踪、被回头解释，那么它最好先经过 PR；如果它只是一次很小的个人动作，才更适合直接推。**

所以对“每加一个功能该不该走 PR”这个问题，我现在的答案是：

**一般应该走，尤其当你已经在意 release notes、changelog 和协作可追踪性的时候。**

但如果再补一句我现在更在意的话，那就是：

**真正让工作流变清晰的，不只是“先开 PR”，而是“把 feature 拆成合适的 PR 切片，再让版本按阶段演进”。**
