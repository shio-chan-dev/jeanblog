> 核心结论：如果一个版本要交给前端、测试或下一个协作环节继续验证，我现在更倾向于给它一个新的 alpha tag，并配套发 GitLab release。它很适合做“交付基线记录层”，但不适合替代 issue、正式文档和 commit。

## 目标读者

- 经常要把版本交给前端、测试、产品或运维继续验证的工程师
- 当前还主要靠群消息、口头同步或 commit hash 传递版本的人
- 正在评估 tag / release 是否值得纳入日常工作流的小团队负责人

## 背景与动机

最近我在一个真实项目里，连续做了两件以前并不会主动做的事：

- 给当前生产基线补了一个正式 release
- 给当前联调基线补了一个 alpha release

更具体一点，这次我的做法是：

- 用正式 tag 表示当前生产版本
- 用 `alpha` tag 表示当前联调版本
- 把 API 变更、字段调整、迁移建议和交接说明写进 GitLab release
- 群里只发短通知和链接

这套做法的比较对象，不是成熟的大厂 release engineering 体系，而是更常见的小团队默认动作：

- 打一个 tag，或者甚至不打
- 群里发一段说明
- 需要时再回头翻聊天记录、commit、接口文档

这个默认动作不是不能用。但只要协作链条开始变长，问题就会慢慢出现：

- 哪个版本才是“这次要测的那个版本”？
- 群里那段说明到底对应哪个 commit？
- 生产基线和测试基线怎么区分？
- 一周之后回看，谁还记得 alpha.1 和 alpha.2 之间到底差了什么？

我这次写这篇文章，不是为了教别人怎么操作 GitLab，而是想把我的判断写清楚：

**对“跨环节交付”这件事来说，alpha tag + GitLab release 值不值得进入日常工作流？**

## 核心概念

先把几个容易混淆的概念拆开。

### 1. Tag

tag 的作用是把一个版本点钉在某个 commit 上。它回答的是：

- 这个版本对应哪一个代码状态？

它解决的是“版本点存在不存在”的问题，不负责解释“这个版本给谁用、改了什么、该怎么看”。

### 2. Release

release 更像 tag 上面的一层说明书。它回答的是：

- 这个版本是什么定位？
- 这次变更要让谁看？
- 哪些点需要特别注意？

它不是代码本身，但它能显著降低“这个版本到底是什么意思”的沟通成本。

### 3. 群消息

群消息适合通知，不适合存档。它的优势是快，劣势也很明显：

- 容易被后续消息冲掉
- 难和具体版本点稳定绑定
- 不适合作为几天之后还要回看的交接依据

### 4. Source of Truth

这是我这次最想强调的边界。

在我的工作流里：

- issue 负责记录为什么做、范围是什么、验收标准是什么
- 正式文档负责记录可持续对接的契约
- commit 负责记录一次代码变更的上下文
- release 负责记录“这次交出去的版本基线是什么”

换句话说，**release 是交付记录层，不是新的权威知识库。**

## 我原本期待它解决什么

在实际尝试之前，我对“alpha tag + release”这套做法的预期其实很朴素。

### 1. 我希望它能明确区分“生产版”和“测试版”

以前团队里经常只有一句话：“这个版本你们可以测了。”

问题是，这句话本身不够精确。尤其当项目同时存在生产版本、开发分支、联调分支时，没有稳定版本号就容易混淆。

### 2. 我希望它能替代一部分群聊里的交接说明

我不想每次都把完整变更点只写在聊天工具里。因为聊天工具的本质不是版本档案。

### 3. 我希望它能给“下一环节的人”一个更清晰的入口

前端、测试、产品，或者未来回头看的自己，最好都能从一个稳定链接进入，而不是从群里开始考古。

## 实际使用后我看到了什么

真正做完这一轮之后，我的看法比一开始更明确了。

### 1. alpha tag 最大的价值不是“版本看起来更专业”，而是“交付边界更清楚”

一旦我把某个联调版本明确标成 `alpha.1`，这件事的含义就变了：

- 它不再只是“当前分支的一次提交”
- 它开始变成“可以交给别人继续验证的一次基线”

这个变化非常重要。因为很多团队的问题，不是不会写代码，而是没有把“内部开发状态”和“对外交接状态”分清楚。

### 2. release 的价值不在于多一个页面，而在于多了一层稳定说明

tag 只能告诉你“版本点在这里”，release 才告诉你“为什么这个版本值得被单独拿出来看”。

这次最直观的体感是：把 API 字段变化、兼容说明、迁移建议写到 release 里之后，群消息终于变回了通知，而不是承载全部上下文的地方。

### 3. 维护成本比我原来想的小

我原来担心 release 会变成一种“每次都要额外写很多废话”的流程负担。

但实际试下来，只要你明确 release 的服务对象是“下一环节的接手人”，它的结构反而很稳定：

- 这是正式版还是测试版
- 这次主要改了什么
- 对接方应该注意什么
- 当前版本和其他版本的关系是什么

它不是写长文档，而是在补一个交付面缺失的说明层。

## 它适合做什么

这里我按几个真正影响判断的维度来比较。

### 1. 追踪性：它比“群里发一段话”强得多

我的旧做法是：

- 群里发一段说明
- 有时打 tag，有时不打

这种方式短期内很快，但最大的问题是说明和代码状态没有稳定绑定。

而 alpha tag + release 的组合，至少把下面三层关系固定住了：

- release 对应哪个 tag
- tag 对应哪个 commit
- 这次交接说明对应哪个版本点

我的判断是：**只要版本要交给别人继续验证，追踪性就是 release 胜出。**

### 2. 团队协作适配度：它更适合“下一个人接着干”

群消息更像“我现在告诉你一声”。

release 更像“我把这次要交给你的版本打包说明好了”。

这两者的服务对象不一样。

如果只是自己记一下，群消息当然更快。  
但如果是前端要联调、测试要验收、运维要识别生产版本，release 更像一个稳定交接面。

我的判断是：**它尤其适合跨角色协作，不只是适合“我自己回头看”。**

### 3. Source-of-truth 适配度：它适合记录交付，不适合替代正式资料

这是我最在意的边界。

我不会因为开始写 release，就把 issue、API 文档、commit message 的职责并过去。

在我的工作流里：

- issue 仍然负责“为什么做”
- API 文档仍然负责“怎么对接”
- commit 仍然负责“这次代码改了什么”
- release 负责“这次给别人什么版本去看”

我的判断是：**release 非常适合做交付记录，但它不是新的 source of truth。**

### 4. 维护成本：有成本，但比返工解释更便宜

这套流程不是零成本。

你至少要多做几件事：

- 给版本命名
- 判断这次是否值得升一个新 alpha
- 写一段 release 说明

但和反复解释下面这些问题相比，我觉得这个成本是值得的：

- “你说的那个版本是哪一个？”
- “这个接口变化到底从什么时候开始算？”
- “alpha.1 和 alpha.2 差别是什么？”

我的判断是：**只要项目已经进入多人协作，这个维护成本通常低于反复补解释的成本。**

## 它不适合做什么

我愿意把它纳入工作流，不代表我觉得它适合所有场景。

### 1. 不适合每次本地修一点东西就升一个 alpha

如果改动还停留在你自己本地，没人要接手验证，也没有形成新的交付基线，那通常只 commit 就够了。

否则会出现一种新的混乱：

- alpha 太多
- 每个 alpha 都很短命
- 团队开始不再认真看 release 说明

### 2. 不适合替代 issue、文档和 commit

如果团队开始把 release 当成唯一说明入口，很快就会出现边界漂移：

- issue 不写清楚范围
- 正式文档不及时更新
- commit 只剩下“fix stuff”

这不是 release 的问题，而是把它用错位置了。

### 3. 不适合在 tag 语义混乱的团队里直接照搬

如果团队内部连这些都没约定清楚：

- alpha 是什么
- rc 是什么
- 正式版是什么
- 谁可以宣布一个版本进入下一阶段

那先补语义，再补 release，比直接创建很多 tag 更重要。

## 和我当前工作流的边界

这次实践之后，我给自己定下来的边界大概是这样的。

### 1. 群消息是通知层，不是存档层

群里仍然要发消息，但我更希望它只做两件事：

- 通知版本已经可用
- 丢出 release 链接

而不是把完整变更点永远留在群聊里。

### 2. release 是交付层，不是知识层

release 应该回答：

- 这次交付的是哪个版本
- 它是什么定位
- 接手人最应该注意什么

但它不应该变成替代 API 文档和 issue 的第二套说明系统。

### 3. alpha 只在“下一环节要接手”时增长

这是我这次最确定的一条实践规则：

- 只有当一个新版本要交给下一环节继续验证时，我才会发新的 `alpha.N`
- 如果只是自己继续改，没有形成新的交付基线，我不会为了“看起来规范”而机械升版本

这条边界一旦明确，alpha tag 就不会泛滥。

## 给读者的最小评估方法

如果你也在判断这套做法要不要纳入自己的工作流，我建议先问自己三个问题。

### 1. 这个版本是不是要交给别人继续验证？

如果答案是否，通常不需要新 alpha。  
如果答案是是，继续往下看。

### 2. 这次变化是不是值得被单独解释？

例如：

- API 有调整
- 行为边界有变化
- 对接方需要迁移说明
- 生产基线和测试基线需要显式区分

如果这些都没有，单纯打个 tag 未必值得。

### 3. 几天之后，你还希望别人知道这次版本为什么存在吗？

如果你希望答案是“知道”，那说明群消息大概率不够，release 更合适。

我自己的简单规则是：

- 只在本地继续开发：`commit`
- 形成新的联调基线：`alpha + release`
- 准备上线前集中验收：`rc + release`
- 生产正式部署：正式版本号 + 正式 release

## 解释与取舍

这套做法不是“更先进”，而是更适合我当前面对的问题。

我真正要解决的不是 Git 功能不够，而是这几个现实问题：

- 交付对象不止我自己
- 版本边界需要区分生产和测试
- 群消息不适合承载长期交接说明
- 我希望以后回看版本时，不用重新从聊天记录拼上下文

在这个问题设定下，alpha tag + GitLab release 的确比“只在群里说一声”更合适。

但如果你的项目仍然是：

- 单人开发
- 没有真正的跨环节交接
- 版本变化也没有形成稳定对外交付面

那它的收益会小很多，甚至可能只是增加流程感。

## 常见问题与注意事项

### 1. 前端或测试提了一个 bug，修完之后要不要直接发 `alpha.2`？

我的答案是：**如果修完之后要把这个新基线重新交给他们继续测，就发。**

关键不是“有没有改代码”，而是“有没有形成新的对外交付基线”。

### 2. 已经发出去的 `alpha.1` 要不要直接改掉？

我不建议。

一旦一个版本已经被别人拿去测了，后续修复更适合发新的 `alpha.2`，而不是重写旧 tag 的含义。

### 3. release 说明里应该写什么？

我的经验是，只写接手人真正关心的东西：

- 版本定位
- 主要变化
- 对接注意点
- 和正式版 / 上一个测试版的关系

不要把它写成流水账，也不要把内部实现细节全部堆进去。

### 4. 我没有验证什么？

我这次没有验证更重的自动化流程，例如：

- 自动生成 changelog
- 自动挂 milestone
- 自动附加 release asset

所以我的结论只覆盖“手动维护 alpha tag + GitLab release 是否值得”这一层。

## 最佳实践与建议

- 把群消息当通知层，不要当长期交接存档层
- 把 release 当交付层，不要当 source of truth
- 只在形成新交付基线时递增 `alpha.N`
- 正式版和测试版的 release 都要明确写定位
- 对前端或测试的 release 说明，优先写契约变化和迁移建议，不要堆内部实现细节

## 结论

这次真实实践之后，我的结论已经比较稳定了：

**alpha tag + GitLab release 很适合做“跨环节交付基线”，但不适合被误用成“新的权威资料系统”。**

如果你已经开始遇到这些问题：

- 版本边界不清楚
- 群消息承担太多上下文
- 前端、测试和后端对“该测哪个版本”理解不一致

那我认为它值得进入你的日常流程。  
但如果你只是自己本地继续开发，还没有真正发生交接，那就没必要为了规范而规范。
