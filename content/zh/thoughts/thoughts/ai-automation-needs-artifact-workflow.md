> 核心观点：AI 不是只能深度参与写代码，而是代码领域已经提前准备好了让 AI 深度参与的基础设施。很多领域不是没有判断好坏的标准，而是没有把标准绑定到每个中间版本上。只要其他领域也建立起版本控制、评价标准、可验证增量、checkpoint、review 和回滚机制，AI 就能以类似写代码的方式介入教程、文章、小说、视频、3D 模型等复杂作品。文字领域甚至可能因此比代码领域释放出更强的 AI 能力，因为语言本来就是大模型最亲近的介质。

## 问题

现在 AI 写代码的介入度明显高于很多其他领域。

一个工程师可以让 AI：

- 读代码库
- 制定实现计划
- 修改多个文件
- 跑测试
- 修复报错
- 拆 commit
- 写 PR 描述

但如果换成写小说、写文章、做视频、做 3D 模型，AI 的使用方式经常还停留在比较浅的一层：

- 帮我想几个标题
- 帮我润色一段话
- 帮我生成一个脚本
- 帮我出几张参考图
- 帮我给这个模型一点建议

差别不在于这些领域不需要 AI，也不在于模型完全不会做这些事。

真正的差别是：**代码领域有一整套让 AI 可以安全推进工作的结构，而很多创作领域还没有把自己的判断标准工程化。**

## 为什么代码领域最适合 AI 自动化

代码领域天然有几个关键条件。

第一，代码有版本控制。

每一次修改都可以被记录下来。你可以看 diff，可以看 commit message，可以回滚，可以比较两个版本之间到底发生了什么。

第二，代码有可验证反馈。

测试、编译、类型检查、lint、运行结果，都会告诉你这次修改是否破坏了系统。AI 不需要只靠“感觉不错”继续往下写，它可以拿到明确反馈。

第三，代码可以拆成增量。

一个大功能可以拆成：

```text
spec -> plan -> build -> test -> review -> simplify -> commit
```

每一个 slice 都可以独立完成、独立验证、独立提交。

第四，代码有清晰的依赖关系。

数据库、接口、业务逻辑、前端、测试，谁依赖谁，通常可以被分析出来。只要依赖图清楚，AI 就知道先做什么、后做什么。

第五，代码有 rollback。

如果某一步错了，不是整件事报废，而是回到上一个可工作的 checkpoint。

这就是为什么 AI 在代码领域不是简单的“写一段文本”，而是可以像一个工程协作者一样工作。

## 其他领域缺的不是标准，而是被版本化的标准

很多领域其实也有复杂产物：

- 教程是一种复杂产物
- 小说是一种复杂产物
- 长文章是一种复杂产物
- 视频是一种复杂产物
- 3D 模型是一种复杂产物
- 课程、演讲、设计稿、研究笔记也都是复杂产物

它们的问题不是不能拆，也不是没有标准。

文章当然有标准：观点是否清楚，段落是否服务主论点，证据是否足够，读者是否能跟上。

小说当然有标准：人物动机是否成立，冲突是否升级，场景是否推动情节，伏笔是否回收。

视频当然有标准：前 30 秒是否建立观看理由，镜头顺序是否自然，声音和画面是否互相支撑。

3D 模型当然有标准：轮廓是否成立，比例是否正确，拓扑是否支持变形或渲染，材质和光照是否服务用途。

真正缺的是：这些标准很少被绑定到具体的中间版本上。

在代码里，一个 commit 会告诉你：

```text
这次改了什么
为什么改
怎么验证
如果错了怎么回退
```

但在很多创作任务里，中间状态经常只是散落的草稿、聊天记录、导出文件和脑子里的判断。标准存在，但没有变成可执行的 checkpoint。

这就导致 AI 很难稳定进入。

比如写文章时，我们经常这样要求 AI：

```text
帮我写一篇关于 X 的文章。
```

这对 AI 来说太像“一次性生成最终代码”。它没有明确的 dependency graph，没有阶段 checkpoint，没有验收标准，也没有版本历史。

结果就是：

- 它可能结构上看起来完整，但核心论证是散的
- 它可能语言流畅，但没有真正回答问题
- 它可能越写越长，但每一段都不可验证
- 你很难知道应该从哪里改，也很难把修改过程沉淀下来

这不是 AI 不会写文章，而是文章缺少类似软件工程的生产结构：没有版本化的中间产物，没有每一步的验收标准，也没有能让 AI 继续接力的历史记录。

## 可以迁移的不是代码语法，而是工程思想

真正可以跨领域迁移的，不是“把所有东西都变成代码”，而是软件工程里这些思想：

```text
目标/读者/场景
-> 压缩成一个小而完整的核心模型
-> 拆依赖图
-> 拆可验证任务
-> 每次只推进一个 checkpoint
-> review
-> simplify
-> freeze / commit
```

这套结构可以叫做 artifact workflow。

这里的 artifact 指任何复杂作品，不只是代码。

关键是把“作品”看成一个可以被持续构建的对象，而不是一次性生成的结果。

更直接一点说：

```text
git 不是重点，版本化思想才是重点。
test 不是重点，可评价标准才是重点。
commit 不是重点，阶段性判断记录才是重点。
refactor 不是重点，完成后再简化结构才是重点。
```

代码领域只是把这些东西做得最显性，所以 AI 最先在那里进入深水区。

## 从代码工作流抽象出来的通用框架

我现在越来越倾向于把复杂作品分成几个阶段。

### 1. Spec：定义作品承诺

先回答：

- 这个作品给谁看
- 它解决什么问题
- 它最终要让读者、观众或使用者获得什么
- 哪些东西不在本次范围内

在代码里，这是需求和接口。

在文章里，这是中心论点。

在小说里，这是主题、冲突和人物承诺。

在视频里，这是观众看完以后应该理解、感受到或记住什么。

### 2. Plan：拆依赖图和任务

不要直接开始写正文或做成品。

先拆：

- 哪些概念必须先讲
- 哪些场景必须先建立
- 哪些例子可以作为贯穿线
- 哪些任务可以独立完成
- 哪些任务必须依赖前一个 checkpoint

代码领域叫 implementation plan。

创作领域也应该有类似的 artifact plan。

一个好的 artifact plan 不只是大纲，而应该像工程计划一样包含：

- 依赖图：哪些概念、场景、角色、镜头、形体必须先出现
- 任务列表：每个任务只推进一个可检查的切片
- 验收标准：这个切片怎样才算完成
- 验证方式：谁来检查，用什么标准检查
- checkpoint：什么时候冻结，冻结后给下一步什么基础

这也是我觉得代码领域的 plan 结构可以迁移到写作和创作领域的原因。

### 3. Build：每次只推进一个可验证切片

不要一次性让 AI 写完整篇文章或完整个视频。

更好的方式是：

```text
Task 1: 写清楚问题场景
Task 2: 写出最小模型
Task 3: 写第一个例子
Task 4: 写第一段论证
Task 5: 做一次 review
Task 6: simplify
```

每一步都要能回答：

- 这一步解决什么压力
- 修改前有什么缺口
- 这一步改了什么
- 怎么验证它变好了
- 现在冻结成什么状态
- 下一步还缺什么

如果是文章，这一步可能不是“写完整篇”，而是：

```text
Task 1: 写真实问题场景
验收：只说明谁遇到问题、为什么痛苦、外部看到什么，不提前讲解决方案。

Task 2: 写核心观点
验收：一句话能说清，后续每一节都能回扣。

Task 3: 写第一个论证切片
验收：有例子，有推理，不只是观点堆叠。
```

如果是小说，这一步可能是：

```text
Task 1: 确定主角欲望和阻力
验收：角色想要什么、为什么现在要、失败代价是什么。

Task 2: 写第一场冲突
验收：场景结束时局势发生变化，而不是只介绍设定。
```

如果是视频，这一步可能是：

```text
Task 1: 写 30 秒开场钩子
验收：观众知道为什么要继续看。

Task 2: 做粗剪结构
验收：每个镜头都有信息功能，没有只负责填时间的片段。
```

如果是 3D 模型，这一步可能是：

```text
Task 1: blockout 大形
验收：silhouette 在远处也能识别目标对象。

Task 2: 做二级形体
验收：比例和功能结构成立，不提前陷入材质细节。
```

### 4. Review：不是最后挑错，而是检查链路

代码 review 不只是看代码能不能跑，还看：

- 设计是否合理
- 边界是否清晰
- 是否引入了不必要复杂度
- 是否破坏了已有行为

文章、小说、视频、3D 模型也一样。

不同领域的 review 标准不同，但都应该存在。

文章 review 看论证链条。

小说 review 看人物动机、冲突推进、节奏和伏笔。

视频 review 看信息密度、镜头顺序、声音、节奏。

3D 模型 review 看轮廓、比例、拓扑、材质、光照和用途。

### 5. Simplify：先完成，再打磨

代码里有 refactor。

创作里也应该有 simplify。

它不是把作品变短，而是删掉没有承担功能的东西，让核心更清楚。

文章里删掉重复论证。

小说里删掉不推进冲突的场景。

视频里删掉节奏拖慢的镜头。

3D 模型里删掉不服务轮廓和用途的细节。

### 6. Checkpoint：把结构化信息放进历史

这是最容易被低估的一步。

代码领域有 commit message。一个好的 commit 不只是“改了文件”，而是记录：

- 为什么改
- 改了什么
- 有什么影响
- 怎么验证

创作领域也需要类似的 checkpoint。

比如写一篇教程时，正文应该保持自然可读，但 commit message 可以记录：

```text
Pressure:
- 读者还不知道为什么普通函数调用不够。

Naive:
- 当前只展示了 prepare(context); enrich(context)。

Break:
- 调用方仍然必须知道步骤顺序，也没有 step status。

Change:
- 增加 mode -> workflow names 的解释和代码。

Check:
- standard 可以解析出 prepare -> enrich，unknown mode 在执行前失败。

Freeze:
- 这一节冻结为“外部意图变成内部计划”。

Still lacks:
- 还没有 handler registry 和 RunRecord。

Next:
- 下一节讲 workflow name 如何变成函数调用。
```

这样正文给人看，commit 给未来的自己和 AI 看。

当你下一次打开这个项目，AI 不需要猜“这篇文章写到哪里了”，它可以直接从历史里知道每个 checkpoint 的意图。

## 不同领域的“测试”长什么样

跨领域迁移这套方法时，最重要的一点是：不要把代码测试照搬过去，而是找到每个领域自己的验证方式。

代码的测试是：

- unit test
- build
- type check
- lint
- manual run

教程的测试是：

- 读者能不能从零跟着做
- 每一步是否先有压力再有解法
- 最后代码是否能独立运行
- 每个概念是否被例子验证

文章的测试是：

- 中心论点是否一句话说得清
- 每一节是否支持这个论点
- 有没有跳步
- 有没有只是在堆观点

小说的测试是：

- 角色行为是否有动机
- 冲突是否升级
- 场景是否推动人物或情节
- 伏笔是否有回收

视频的测试是：

- 前 30 秒是否建立观看理由
- 镜头顺序是否自然
- 声音和画面是否互相支撑
- 信息密度是否适合目标观众

3D 模型的测试是：

- silhouette 是否成立
- 比例是否正确
- 拓扑是否支持变形或渲染
- 材质和光照是否服务目标用途

只要一个领域能定义自己的测试，AI 就能开始闭环工作。

## 为什么这会提高 AI 介入度

AI 自动化最怕的不是任务难，而是反馈模糊。

如果你只说“写得更好一点”，AI 很难知道什么叫更好。

但如果你说：

```text
当前 checkpoint 的问题是：论证没有从真实场景开始。
任务：补一个真实场景，只解释谁遇到这个问题、为什么痛苦、外部看到什么。
验收：不引入解决方案，不提前讲框架，不超过 400 字。
```

AI 就能做得非常具体。

代码领域的高自动化，靠的正是这种具体性：

- diff 明确
- 任务明确
- 测试明确
- 回滚明确
- 历史明确

其他领域如果也拥有这些东西，AI 就不再只是“灵感生成器”，而会变成真正的生产协作者。

## 文字领域可能比代码领域更先爆发

这里还有一个反直觉的结论。

代码领域最先被 AI 深度自动化，不一定意味着代码领域是 AI 最擅长的领域。它更像是最早准备好了工作条件的领域。

代码有版本控制、测试、构建、类型检查、回滚机制，所以 AI 可以安全地参与。但代码也有很多硬约束：

- 编译器不会因为表达流畅就放过错误
- 运行环境、依赖版本、框架细节都可能变化
- 一个很小的 bug 可能让整个系统不可用
- 很多质量问题必须靠真实运行才能暴露

文字领域反而不一样。

语言生成、结构重组、观点提炼、风格转换、摘要、反驳、类比、润色，本来就是大模型最自然的能力。文章、教程、脚本、小说这些东西，评价标准虽然不如编译器那么硬，但很多判断本身就是语言理解：

- 这个观点是否清楚
- 这个段落是否服务主论点
- 这个例子是否支撑结论
- 这个角色动机是否成立
- 这个开头是否建立了阅读压力
- 这个结尾是否兑现了承诺

这些问题，AI 不只是能生成答案，也能参与评估、改写、对比和迭代。

所以真正限制 AI 写作的，可能不是模型不会写，而是我们太习惯让它“一次写完”。一次性生成全文时，AI 很容易变成平均化的文本机器；但如果文章也有类似代码的 artifact workflow，它就可以变成一个更强的写作协作者：

```text
读者承诺
-> 核心观点
-> 论证依赖图
-> 段落任务
-> 每段验收标准
-> review
-> simplify
-> checkpoint
```

这时 AI 的能力会更贴近它的本体优势。

它可以同时扮演作者、编辑、反对者、目标读者和结构审查者。它可以快速给出多个论证版本，可以检查一段话是否跑题，可以把抽象观点改写成例子，也可以把松散草稿重组为清晰结构。

代码领域给了我们方法论，但文字领域可能是这套方法论更早爆发的地方。

因为代码里的很多验证需要外部系统，而文字里的很多验证，本身就在语言里。

一旦文章、教程、小说这些文字作品补上版本化、任务化、验收标准和 checkpoint，AI 可能不只是达到“类似写代码”的自动化程度，而是超过它。

## 这不是把创作工业化，而是保护创作

这里可能有一个误解：如果给小说、文章、视频、模型都加上 workflow，是不是会让创作变得机械？

我认为恰好相反。

没有 workflow 时，AI 很容易把作品变成平均化的流水账。因为它只能一次性猜一个完整结果。

有 workflow 时，人反而更能保留主导权：

- 人定义作品承诺
- 人定义审美标准
- 人决定 checkpoint 是否通过
- AI 负责推进局部任务
- 历史记录保留每次取舍

这不是让 AI 替代作者，而是让作者拥有更强的执行杠杆。

## 一旦有了 workflow，AI 就能进入任何领域

这也是我最近意识到的关键点。

AI 进入一个领域，不一定要先“理解这个领域的一切”。它需要的是一个能让它安全工作的外骨骼：

```text
作品目标
-> 领域核心模型
-> 任务分解
-> 每步产出
-> 每步评价标准
-> checkpoint
-> review
-> simplify
-> version history
```

只要这个外骨骼存在，AI 就可以先从局部任务开始工作。

文章不是让 AI 一次生成全文，而是让它完成“补强开头压力”“重写一个论证切片”“删除不服务主论点的段落”。

小说不是让 AI 一次生成整本书，而是让它完成“检查这个场景是否推进冲突”“补一个角色动机”“回收一个伏笔”。

视频不是让 AI 一次生成成片，而是让它完成“重排前三个镜头”“检查旁白和画面是否重复”“为这一段补 B-roll 需求”。

3D 模型不是让 AI 一次交付最终资产，而是让它参与“检查 blockout 轮廓”“给拓扑问题做标注”“根据用途提出材质简化方案”。

这时 AI 的角色就从“生成器”变成了“迭代协作者”。

区别非常大。

生成器只负责给你一个结果。

迭代协作者可以在版本之间持续工作，可以根据评价标准修正，可以读历史，可以知道当前 checkpoint 到底冻结了什么。

## 结论

代码领域并不是因为“代码特殊”才最先被 AI 自动化。

更准确地说，代码领域早就拥有 AI 需要的工作条件：

- 版本控制
- 评价标准
- 可验证增量
- 依赖图
- checkpoint
- review
- simplify
- rollback

这些东西让 AI 可以在不完全理解整个世界的情况下，安全地推进一个局部任务。

如果教程、文章、小说、视频、3D 模型这些领域也建立类似的 artifact workflow，它们就会得到类似写代码领域的 AI 自动化能力。

其中，文字领域可能会更快得到回报。因为 AI 本来就擅长语言，只是过去缺少足够清晰的版本控制、任务边界和评价标准来约束这种能力。

未来真正重要的能力，可能不是“会不会让 AI 生成内容”，而是：

**能不能把一个领域的复杂作品，改造成 AI 可以增量构建、验证、回滚和长期协作的对象。**

换句话说，AI 自动化的边界，不只取决于模型有多强，也取决于我们能不能为某个领域建立起足够好的作品工作流。
