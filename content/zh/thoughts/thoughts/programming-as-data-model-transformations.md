> 核心观点：我越来越倾向于把写程序看成“数据模型之间的转换”，而不是“把代码一行一行拼出来”。这个方法很适合算法、数据结构、状态管理和业务流程建模；但它不是银弹，不能替代语法熟练度、API 经验、测试反馈和工程约束。

## 目标读者

- 正在学算法和数据结构，但经常被代码细节带跑的人
- 写业务代码时容易陷入“先写再说”，但后面又发现状态混乱的人
- 使用 AI 辅助写代码，希望自己仍然能解释核心逻辑的人

## 背景与动机

这次触发点是 Trie。

一开始看这段代码时，我的问题不是“语法看不懂”，而是：

```python
if ch not in node.children:
    node.children[ch] = TrieNode()
node = node.children[ch]
```

这些节点到底从哪里来？
`children` 是字典还是集合？
`node = node.children[ch]` 到底在移动什么？
`is_end` 又为什么需要？

如果只盯着代码，很容易觉得它是一套固定写法。
但一旦把数据模型画出来，事情就变清楚了：

```text
字符串： a   p   p   l   e
        ↓   ↓   ↓   ↓   ↓
Trie：  root -> a -> p -> p -> l -> e
```

程序不是在“背模板”，而是在用字符串里的字符驱动指针在树上一步步移动。
插入时，缺边就补边；查询时，缺边就失败；完整单词查询最后还要检查 `is_end`。

这让我意识到：很多程序都可以先不从代码开始，而是先从数据模型开始。

## 核心概念

我现在会把程序拆成四个东西：

```text
Data model: 数据长什么样
State: 当前维护什么状态
Transition: 每一步如何改变状态
Invariant: 每一步之后什么必须保持为真
Answer: 最后从哪里读出答案
```

以 Trie 为例：

```text
Data model:
- 一棵前缀树，每个节点代表一个前缀

State:
- node 指向当前前缀对应的节点

Transition:
- 读取一个字符 ch
- 如果是插入，缺少 ch 这条边就创建
- 然后 node 移动到 children[ch]

Invariant:
- 处理完 word[:i] 后，node 指向这个前缀对应的节点

Answer:
- search 看 node 是否存在并且 node.is_end 为 True
- starts_with 只看 node 是否存在
```

这个模型比“记住几行代码”更稳定。

## 我原来倾向的写法

原来的方式更像是：

1. 先看答案代码
2. 逐行解释语法
3. 模仿结构写一遍
4. 遇到变体再回头改

这不是完全没用。
如果目标只是快速过一题，模仿代码可以很快。

问题是，一旦条件变化，理解就会断：

- 如果要支持删除，节点上要加什么？
- 如果要统计某个前缀出现几次，状态放在哪里？
- 如果字符集不是小写字母，`children` 应该怎么表示？

没有数据模型时，变体会变成“另一个模板”。
有数据模型时，变体只是“在节点上增加字段或改变转换规则”。

## 我现在更认可的写法

现在我更认可这个顺序：

```text
先画数据模型
-> 写状态和不变量
-> 写伪代码或语义注释
-> 翻译成代码
-> 用测试和 trace 校验
```

写 Trie 的插入时，可以先写注释：

```python
def insert(self, word: str) -> None:
    # start from the empty prefix

    # for each character in the word:
        # if the current prefix has no edge for this character:
            # create a child node for that character

        # move to the child node
        # now node represents the prefix including this character

    # after all characters are consumed:
        # mark the current node as the end of a complete word
```

再翻译成代码：

```python
def insert(self, word: str) -> None:
    node = self.root
    for ch in word:
        if ch not in node.children:
            node.children[ch] = TrieNode()
        node = node.children[ch]
    node.is_end = True
```

关键不是注释多，而是注释先表达“模型如何变化”。
等模型熟了，注释自然可以减少。

## 它适合什么

这个方法很适合三类场景。

第一，算法和数据结构。

比如滑动窗口：

```text
Data model: 数组或字符串
State: [l, r] 窗口和频次表
Transition: r 右移加入元素，l 右移移除元素
Invariant: 频次表必须等于当前窗口真实内容
Answer: 满足条件时更新最优值
```

第二，业务流程。

很多业务代码本质也是状态转换：

```text
订单草稿 -> 已提交 -> 已支付 -> 已发货 -> 已完成
```

真正要想清楚的是：

- 每个状态允许哪些操作
- 哪些字段随状态改变
- 哪些转换不允许发生
- 失败时是否回滚

第三，前端状态管理。

页面状态也可以这样看：

```text
URL 参数 -> 查询条件状态 -> 请求状态 -> 列表数据 -> 选中项/弹窗状态
```

如果模型不清楚，组件很容易堆成一团。

## 它不适合什么

这个方法不是万能的。

它不能替代语法和 API 熟练度。
比如你知道 Trie 的模型，但不熟悉 Python 字典，还是会卡在：

```python
if ch not in node.children:
```

它也不能替代测试。
模型再清楚，也可能漏边界：

- 空字符串怎么办？
- 重复插入怎么办？
- 删除时是否要清理孤立节点？

它也不适合把所有代码都过度抽象成模型。
有些代码只是调用库、处理格式、拼装参数，直接写清楚就够了。
如果为了“模型化”而引入复杂概念，反而会变慢。

## 与当前工作流的边界

我不打算把这个方法变成一套沉重流程。

我的边界是：

- 学新数据结构时，必须先画模型
- 写核心逻辑时，先写语义注释或伪代码
- 写业务流程时，先列状态和转换
- 写简单胶水代码时，不强行建模

也就是说，它应该成为思考方式，而不是仪式。

尤其在使用 AI 时，这个方法更重要。
如果直接让 AI 写代码，我可能得到一段能跑的实现，但不一定知道它为什么对。
如果我先写出数据模型、转换规则和不变量，再让 AI 检查或补代码，我仍然保留了主导权。

## 一个最小自检方法

写代码前，可以问自己五个问题：

```text
1. 输入是什么模型？
2. 中间状态是什么？
3. 每一步怎么改变状态？
4. 每一步之后什么必须保持为真？
5. 最后从哪里读出答案？
```

如果这五个问题答不出来，直接写代码大概率会变成试错。

如果能答出来，代码通常只是翻译。

## 解释与取舍

这个方法的收益是稳定。

它让代码不再只是语句序列，而是有一个可解释的结构。
当代码出错时，也更容易定位：

- 数据模型错了？
- 状态没有更新？
- 转换规则漏了？
- 不变量被破坏了？
- 答案读取位置错了？

代价是前期会慢一点。
尤其刚开始练习时，写模型、写伪代码、写 invariant 会觉得啰嗦。

但这个慢不是浪费。
它是在把“隐含思考”显式化。
等熟练之后，很多模型会自动出现在脑子里，代码反而更快。

## 常见问题与注意事项

### 是不是每行代码都要先写注释

不是。

真正值得写的是语义注释，而不是语法复述。

差的注释：

```python
# assign root to node
node = self.root
```

好的注释：

```python
# start from the empty prefix
node = self.root
```

前者只是翻译代码，后者说明模型含义。

### 是不是所有程序都能变成模型转换

大部分可以，但粒度不同。

算法题通常是清晰的模型转换。
业务代码可能是状态机、数据流或权限规则。
胶水代码可能只是把一个库的返回值转成另一个库需要的参数。

不要为了统一而过度设计。

### AI 在这里应该怎么用

我更愿意让 AI 做三件事：

- 检查我的数据模型有没有缺口
- 挑战我的不变量是否成立
- 根据我写好的伪代码补出实现或测试

我不希望 AI 直接跳过模型给我最终代码。
那样短期快，长期会削弱理解。

## 最佳实践与建议

- 学一个新结构时，先画一张图
- 给核心变量写一句“它代表什么”
- 给循环写一句 invariant
- 先写语义注释，再翻译成代码
- 用一个小 trace 检查状态是否真的按预期变化
- 最后删掉低价值注释，保留帮助理解模型的注释

## 结论

我现在的判断是：写程序时先想数据模型和转换规则，是一种值得长期保留的基本功。

它不会让所有问题自动变简单，但能让很多问题从“背代码”变成“解释状态如何变化”。
尤其在学习算法和使用 AI 辅助编程时，这个方法能帮助我保持理解，而不是只得到答案。
